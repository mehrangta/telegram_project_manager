- Run the complete repository suite: `uv run python -m unittest discover -s tests`
- Assert access-mode compatibility in tests: `/ask` uses `Sandbox.read_only`, `/code` planning uses `Sandbox.read_only`, `/code` implementation uses `Sandbox.workspace_write`, and only `/do` uses `Sandbox.full_access`.
- Assert `/do` passes the startup-resolved bot working directory to Codex and never reads active repository, branch, or local-cache settings.
- Assert private-chat enforcement occurs before queueing or invoking Codex, including for registered admins sending `/do` from a group or forum topic.

## Risks

- `Sandbox.full_access` removes filesystem restrictions and can enable destructive host-level and network side effects. Restricting the command to registered admins in private chats reduces exposure but does not protect against a compromised admin account or unsafe request.
- Using the bot service working directory makes behavior dependent on the deployment service's configured current directory. Deployment documentation and tests must make that root explicit, although full-access Codex can still reach paths outside it.
- Jobs may leave partial, non-idempotent effects when Codex fails, times out, the bot restarts, or shutdown interrupts execution. Deliberately avoiding automatic recovery prevents duplicate execution but requires manual inspection before resubmission.
- Serial execution prevents two `/do` agents from racing each other, but jobs can still conflict with external operators, system services, scheduled tasks, or `/code` jobs running concurrently.
- Codex output and errors can contain credentials or sensitive host data. Pattern-based redaction and developer instructions reduce accidental Telegram disclosure but cannot identify every secret format.
- The pinned `openai-codex==0.1.0b3` SDK is beta. Its current `Sandbox.full_access` value maps to `dangerFullAccess`; regression tests must detect any future change to sandbox or approval semantics.
