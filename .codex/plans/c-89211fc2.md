# Codex plan for mehrangta/telegram_project_manager#3

Job: `c-89211fc2` · Revision: 1

Add an admin-triggered `/do <job>` command that forwards the unmodified job text to the existing Codex SDK using `Sandbox.full_access`, runs asynchronously with bounded concurrency, and replies to the originating Telegram message with the final plain-text result or a sanitized failure. Keep the existing structured `/ask` and `/code` interfaces backward-compatible. Two material decisions remain: the Codex working directory and whether this host-level command is private-chat-only.

## Steps

Share thread start/resume, progress streaming, timeout, interruption, completion-status checking, and active-turn cleanup internally, but omit `output_schema` and JSON parsing for text turns and return the final agent message verbatim. Continue using `ApprovalMode.auto_review`; the caller must explicitly pass `Sandbox.full_access`. Preserve all existing structured callers and tests unchanged.
   - Likely files: `src/telegram_project_manager/bots/code_manager/codex_sdk.py`, `tests/test_code_manager.py`
2. **Implement do command handling** — Create a `DoManager` that recognizes `/do` and `/do@BotName`, independently enforces the admin role like the other managers, and treats everything after the command as the plain job text without issue, plan, repository, or schema wrapping. Return a usage reply tied to the source message when the job is empty. Apply the selected chat-scope policy from the questions before submitting the task; do not add confirmation because the issue explicitly requests direct dispatch.
   - Likely files: `src/telegram_project_manager/bots/do_manager/__init__.py`, `src/telegram_project_manager/bots/do_manager/commands.py`
3. **Run bounded full-access jobs** — Create a `DoService` modeled on the lightweight asynchronous `/ask` service, but without preparing a Git worktree. Reject submissions before acknowledgment when the outstanding queue is full, generate an internal `d-xxxxxxxx` correlation ID, acknowledge the originating message, and schedule the Codex turn under a single-job semaphore to prevent concurrent unrestricted agents from conflicting on host state. Use the configured Codex code model, high reasoning effort, a two-hour timeout, `Sandbox.full_access`, the selected working directory, and minimal developer instructions that execute the supplied job while forbidding credential disclosure in the final response. Reply in the original chat/topic and to the original Telegram message; pass success text through the normal Telegram formatter/4
   - Likely files: `src/telegram_project_manager/bots/do_manager/service.py`, `src/telegram_project_manager/bots/do_manager/prompts.py`, `src/telegram_project_manager/platform/responses.py`
4. **Handle failures and lifecycle** — Record queued, succeeded, failed, and interrupted audit events without persisting the raw job or model response. Sanitize exception text and obvious API-key or bearer-token patterns before Telegram delivery and logging. On service shutdown, interrupt active Codex turns before cancelling and gathering background tasks. Do not automatically recover `/do` jobs after restart because arbitrary full-access work may be non-idempotent; report interruption through logs/audit and require the admin to submit again. Log Telegram delivery failures without rerunning a successfully completed task.
   - Likely files: `src/telegram_project_manager/bots/do_manager/service.py`, `src/telegram_project_manager/platform/storage/db.py`

Add `/do <job>` to bot help and README command/workflow documentation, explicitly labeling it admin-only, unrestricted, non-persistent, and subject to the selected working-directory and chat-scope policy. No database migration or new configuration key is needed unless the configurable-directory option is selected.
   - Likely files: `src/telegram_project_manager/main.py`, `src/telegram_project_manager/bots/commit_manager/commands.py`, `README.md`, `src/telegram_project_manager/platform/config.py`

## Risks

- `Sandbox.full_access` removes filesystem restrictions and can permit destructive host-level or network side effects; Telegram admin authorization is therefore the primary security boundary.
- A successful or interrupted job may leave partial, non-idempotent side effects. Automatic restart recovery would risk repeating actions, so jobs should remain ephemeral and require explicit resubmission.
- The working directory materially changes what Codex sees first and which repository instructions apply, even though full access permits reaching paths outside that directory.
- Model output or command diagnostics can contain sensitive data. Pattern-based redaction and developer instructions reduce exposure but cannot guarantee that arbitrary secrets are never disclosed.
- Multiple unrestricted agents could race on files, processes, package managers, or remote resources; serial execution reduces but does not eliminate conflicts with external operators.
- The pinned `openai-codex==0.1.0b3` SDK is beta. Tests should lock in the current `Sandbox.full_access` and plain-response behavior so an SDK upgrade cannot silently change access semantics.
