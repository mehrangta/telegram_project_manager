# Codex plan for mehrangta/telegram_project_manager#12

Job: `c-bbf901d2` · Revision: 1

Improve the existing automatic pull-request conflict recovery rather than adding a parallel fixer. The current flow already performs guarded Codex-assisted rebases, reruns CI, resumes merge/deploy after restart, and preserves manual recovery controls, but it treats every repeated GitHub conflict report as another failed attempt without recording the PR base revision or distinguishing a moving base, delayed GitHub mergeability data, a merge-time race, and a genuinely unresolved conflict. Implement progress-aware recovery that records the exact head/base pair, waits for stale GitHub state to settle, automatically rebases when the base actually advances, recovers conflicts discovered during the squash-merge command, and stops only on a repeated non-progressing state, an overall recovery deadline, or an existing safety failure. Preserve branch protection, checked-head validation, CI gating, sensitive-path restrictions, force-with-lease pushes, and terminal admin controls.

## Risks

- GitHub may briefly return stale or internally inconsistent `mergeable` and `mergeStateStatus` values after a force-push; bounded polling and exact head/base comparisons are required to avoid both false failures and infinite retries.
- The target branch can advance repeatedly while CI runs. Resetting consecutive attempts on a new base improves automatic recovery, but the persisted overall deadline must remain authoritative so a busy repository cannot keep an operation active indefinitely.
- A merge command can fail for conflicts, policy, authentication, review, or branch-protection reasons. Recovery must be selected from the refreshed PR state rather than matching untrusted CLI error text, and non-conflict errors must retain current failure behavior.
- Force-pushing rebased branches remains destructive if the remote head changes concurrently. Preserve the existing explicit `--force-with-lease=<ref>:<expected-sha>` behavior and checked-head validation without fallback to an unconditional force push.
- Codex conflict resolution can produce semantically incorrect code even when Git conflicts and focused validation pass. Existing file-scope restrictions, sensitive-path blocking, CI gating, maximum conflict rounds, and manual fallback must remain intact.
- Adding snapshot fields and nullable database columns is backward compatible, but all fake `PullRequestSnapshot` constructors and database update allowlists must be updated together to prevent test or runtime initialization failures.
- A no-op rebase must not be treated as successful repair unless Git verifies that the exact fetched base SHA is already an ancestor of the checked head; otherwise the service could trust stale local refs and attempt an unsafe merge.
