# Codex plan for mehrangta/telegram_project_manager#9

Job: `c-09c2a71c` · Revision: 2

Add `/issues` as a read-only command that lists up to 20 open issues for the active chat or forum-topic repository, ordered by most recently updated. Each row shows the issue title and a clickable `#number` linking to GitHub, so the same number can be passed directly to `/code`. Preserve the existing repository allowlist, topic isolation, registered-admin router gate, asynchronous GitHub execution, and error conventions. No database, configuration, or schema changes are required.

## Risks

- `gh issue list --search sort:updated-desc` depends on a GitHub CLI version supporting structured issue output and search sorting; an outdated CLI will surface through the standard `Issues not loaded` failure path and should be caught during deployment validation.
- Constructing `OutgoingMessage` HTML directly bypasses the normal escaping renderer; every dynamic title, repository value, and URL attribute must be escaped before interpolation.
- Plain URLs would trigger the repository’s automatic URL-button generation and could create a large keyboard; linking each issue number inside explicit HTML while setting an empty keyboard avoids that behavior.
- Calling the reader directly from the async handler would block Telegram polling during GitHub latency; the implementation must retain `asyncio.to_thread`.
- GitHub authentication, repository visibility, rate limiting, or transient CLI failures can prevent listing otherwise valid repositories; failures must not clear or mutate active repository settings.
- Changing `CommitManager.__init__` requires updating its single production construction site and any future direct test fixtures; omission would fail at startup rather than degrade only `/issues`.
- The command intentionally returns only the first 20 open issues and offers no pagination or filtering. This bounds Telegram output but means older open issues require GitHub or a future pagination feature.
- The existing router silently ignores all non-admin users before handlers run, so `/issues` will share that behavior even though the command itself is read-only; changing broader access control is outside this issue.
