# Codex plan for mehrangta/telegram_project_manager#9

Job: `c-09c2a71c` · Revision: 1

Add a read-only Telegram command for the active chat/topic repository. The draft assumes the recommended product choices: `/issues`, open GitHub issues only, sorted by most recently updated, and a compact number/title list. Implement the GitHub read behind a dedicated integration service, exclude pull requests returned by GitHub’s issues endpoint, preserve the repository allowlist and topic isolation rules, and make no database or configuration schema changes.

## Steps

1. **Add issue-list integration** — In `integrations/gh/issues.py`, add an immutable issue-summary model and a `GhIssueReader` (or equivalently named read service) accepting `GhRunner`. Define `list_open_issues(repo: str, limit: int = 20) -> list[IssueSummary]`. Request `repos/{repo}/issues?state=open&sort=updated&direction=desc&per_page=100` through `GhRunner.api_value`, require a top-level JSON array, ignore non-object entries and entries containing `pull_request`, validate positive integer issue numbers, normalize titles to one line, cap displayed titles to a fixed safe length such as 120 characters, preserve GitHub ordering, and return at most 20 real issues. Raise a clear `ValueError` for a malformed top-level response while allowing `GhError` to propagate unchanged.
   - Likely files: `src/telegram_project_manager/integrations/gh/issues.py`
2. **Wire the command dependency** — Instantiate the new issue reader from the existing shared `GhRunner` in `run_bot` and inject it into `CommitManager`. Keep the existing `GhIssueExecutor` unchanged for issue creation, avoiding a rename or broader refactor. Update direct `CommitManager` construction sites in tests as required by the new constructor dependency.
   - Likely files: `src/telegram_project_manager/main.py`, `src/telegram_project_manager/bots/commit_manager/commands.py`, `tests/test_commands.py`
3. **Implement active-repo command** — Register `/issues` in `CommitManager.handle`. Resolve the repository with `db.get_scope_settings(message.chat_id, message.thread_id)` so forum topics remain isolated and do not inherit the parent chat repository. If absent, return the established `No active repo for this chat/topic. Admin: /repo set owner/repository` guidance; if not allowlisted, return the established allowlist guidance. Run the GitHub request with `asyncio.to_thread` so network-bound `gh` execution does not block Telegram polling. Catch `GhError` and malformed-response `ValueError`, audit an `issues.list` failure with repository and error text, and return `Issues not loaded.\nReason: ...`. For an empty result, return `No open issues for owner/repo.`; otherwise return `Open issues for owner/repo:` followed by one `- #<n>
   - Likely files: `src/telegram_project_manager/bots/commit_manager/commands.py`
4. **Constrain Telegram output** — Format only issue number and normalized title in the default response; numbers are immediately usable with the existing `/code #123` workflow and omitting per-item URLs avoids generating a large automatic inline-button keyboard. Cap the integration result at 20 issues and each title at 120 characters, then pass the final response through the existing `truncate` helper as a final 4096-character safety guard. Do not add pagination, filters, labels, assignees, persistence, or background jobs unless the product choices below request them.
   - Likely files: `src/telegram_project_manager/bots/commit_manager/commands.py`, `src/telegram_project_manager/platform/responses.py`
5. **Document the interface** — Add `/issues` to both the runtime `/help` text and README command table, describing it as listing open issues for the active repository.    - Likely files: `src/telegram_project_manager/bots/commit_manager/commands.py`, `README.md`
6. **Add integration unit coverage** — Extend GitHub issue tests with a fake `GhRunner` response covering exact endpoint/query construction, recent-update ordering preservation, pull-request exclusion, invalid/non-object entry handling, the 20-item cap, title normalization/truncation, an empty list, malformed top-level JSON, and propagation of `GhError`. Retain all existing issue-creation tests unchanged.

## Risks

- GitHub’s REST issues collection includes pull requests; failing to filter entries containing `pull_request` would violate the command’s issue-only intent.
- A repository with many or very long issue titles can exceed Telegram’s message limit or produce an unwieldy response; fixed result and title caps plus final truncation prevent delivery failures.
- Calling `gh` synchronously inside the async handler would pause polling during slow GitHub responses; the implementation should use `asyncio.to_thread`.
- GitHub authentication, repository visibility, rate limits, or API errors can make a valid active repository unreadable; surface the sanitized `GhError` reason and audit the failure without changing repository settings.
- Adding the read behavior directly to `GhIssueExecutor` would couple listing to its Telegram image-upload dependency; a separate reader avoids that unnecessary compatibility burden.
- Changing `CommitManager.__init__` can break test fixtures or other construction sites; update all constructor call sites while leaving `object.__new__` tests unaffected unless they exercise `/issues`.
- The recommended compact output intentionally omits URLs, labels, assignees, pagination, and closed issues; choosing richer output requires revisiting message-size and keyboard behavior.
