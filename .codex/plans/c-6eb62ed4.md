# Codex plan for mehrangta/telegram_project_manager#8

Job: `c-6eb62ed4` · Revision: 1

Implement an admin-only `/queue` command scoped to the current Telegram chat or exact forum topic. It should aggregate all current Codex-backed workloads—persisted `/code` jobs, in-memory `/ask` questions, and durable `/do` jobs—while separating running work from queued work. The command must use local state only, expose concise presentation-safe metadata, avoid misleading global capacity or position claims, and preserve all existing command and scheduler behavior.

## Steps

Accept bot-qualified forms such as `/queue@ProjectBot`, reject any arguments with `Usage: /queue`, and apply the existing admin permission check. Scope every result by `telegram_chat_id` and exact `telegram_thread_id`, including `IS NULL` semantics for ordinary chats, so private or host-job content cannot appear in another chat or topic. Return `No Codex work is running or queued for this chat.` or the corresponding topic message when empty. Do not display global capacity denominators, numbered queue positions, or estimated start times because the three services have independent concurrency limits and `/do` dispatch can skip a blocked lane.

## Risks

- The three workloads use different schedulers and durability models: `/code` is persisted but scheduled in the bot process, `/ask` is in memory, and `/do` is persisted and executed by a separate worker. The output must keep them in separate sections rather than implying one global FIFO queue.
- The `/do` worker serializes jobs sharing a repository or host lane while allowing different lanes to run concurrently, so displayed ordering cannot be described as an exact execution position.
- Queue snapshots are inherently transient. A job can change state between the code, ask, and do snapshot reads, so the response is a race-safe best-effort view rather than an atomic cross-service transaction.
- Reading `/do` request previews from payload files can race with worker cleanup. Missing, unreadable, symlinked, or invalid files must degrade to a neutral unavailable-preview label without hiding the database job or failing the entire command.
- Question and request text is user-controlled and may contain HTML, newlines, secrets, or excessive content. Normalize and truncate previews, rely on the existing escaping formatter, and never include full raw prompts or filesystem paths.
- Restarting the bot interrupts active `/code` jobs and drops `/ask` entries; restarting the do worker interrupts active `/do` jobs. Roll out during an idle window, restart only the bot to enable the command when possible, and defer the worker restart because this feature requires no worker-side schema or execution change.
- Status collections can drift when future execution phases are added. Keep visibility classifications centralized and test every member so new phases require an explicit queue-display decision.
