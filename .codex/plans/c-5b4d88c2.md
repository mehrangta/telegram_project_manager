# Codex plan for mehrangta/telegram_project_manager#1

Job: `c-5b4d88c2` Â· Revision: 1

The repository already contains the image plumbing needed by Codex: Telegram photos, image documents, and albums become ordered `IncomingAttachment` values; `TelegramBotApi.download_file` retrieves bounded file content; and `CodexSdkAdapter.run_turn` already accepts local image paths and converts them to `LocalImageInput`. However, `/ask` does not exist in the current source, documentation, tests, or visible Git history. This plan therefore assumes the recommended scope: add a one-shot, read-only, repository-aware `/ask <question>` command with optional images. The change is additive, uses `codex_plan_model`, requires no database migration or dependency upgrade, and preserves existing `/issue` and `/code` behavior.

## Risks

- The issue refers to an existing `/ask` command, but no implementation or historical contract is available in this repository. Its conversation model is therefore the only material unresolved product decision.
- Telegram MIME types and declared sizes are untrusted; metadata-only checks would permit spoofed or underestimated files, so downloaded content must be validated independently.
- Images can contain prompt-injection text. Both the question and pixels must remain untrusted inputs under ask-specific developer instructions and a read-only sandbox.
- The pinned `openai-codex==0.1.0b3` dependency is a beta SDK. `LocalImageInput` is already used and tested by `/code`, so this implementation should reuse that boundary rather than upgrade or rely on undocumented behavior.
- Detached worktrees can race other jobs or leak disk after abrupt termination. Repository locking, generated directories, unconditional cleanup, and startup stale-workspace cleanup are required.
- An accepted asynchronous ask is not resumable after a hard restart because the recommended scope adds no persistent ask-job table; the user may need to retry even though stale local files are cleaned.
- Refactoring shared image validation could regress `/issue` error behavior or asset handling. Compatibility tests must lock existing limits and GitHub issue-image behavior before rollout.
- Codex answers can exceed Telegram limits or contain markup, so every completion and error must use existing escaping and truncation.
