# Codex plan for mehrangta/telegram_project_manager#1

Job: `c-6329dd70` · Revision: 1

Extend the existing `/ask` pipeline so a command caption can include Telegram photos, image documents, or albums; validate them with the repository’s established image policy, stage verified files in the temporary read-only ask workspace, pass their local paths through the already-multimodal `CodexSdkAdapter`, and remove them with the job workspace. Preserve the current requirement for a textual question, text-only behavior, permissions, queueing, response schema, and repository-source grounding. No database, configuration, or Codex adapter migration is required.

## Steps

1. **Define ask image contract** — Add an ask-specific image helper that accepts `tuple[IncomingAttachment, ...]`, supports JPEG, PNG, and GIF, and enforces the existing repository policy of at most 10 images, 10 MB per image, and 20 MB total. Provide separate metadata validation for command-time rejection and download-time staging that maps trusted MIME types to deterministic extensions, verifies JPEG/PNG/GIF magic bytes, rechecks actual downloaded sizes, and raises stable `ValueError` messages without exposing Telegram file identifiers or local paths.
   - Likely files: `src/telegram_project_manager/bots/ask_manager/images.py`
2. **Validate and forward attachments** — Update `AskManager.handle` to keep requiring `/ask <question>` in the message text or media caption, validate `message.attachments` before repository lookup and queue acknowledgment, pass the immutable attachment tuple into `AskService.submit`, and update usage text to state that images are optional. Unsupported types, excessive counts, and declared-size violations must return an `OutgoingMessage` replying to the original command and must not create a job or download data.
   - Likely files: `src/telegram_project_manager/bots/ask_manager/commands.py`, `src/telegram_project_manager/bots/ask_manager/images.py`
3. **Stage images for Codex** — Extend `AskService.submit` and `_run` with an `attachments` parameter. After preparing the detached read-only worktree and while holding the existing ask semaphore, download each attachment through `TelegramBotApi.download_file`, write verified bytes in Telegram album order to `<ask-job>/repo/.codex/ask-images/<position>.<extension>`, and pass resolved paths as `image_paths` to `CodexSdkAdapter.run_turn`. Include an `Images: N` line in the queue acknowledgment when images are present. Text-only calls must pass an empty tuple so `CodexSdkAdapter` retains its current plain-text turn input.
   - Likely files: `src/telegram_project_manager/bots/ask_manager/service.py`, `src/telegram_project_manager/bots/ask_manager/images.py`, `src/telegram_project_manager/bots/code_manager/codex_sdk.py`
4. **Harden grounding and cleanup** — Update ask developer instructions to identify attached images as untrusted user context whose visible instructions must not override the task, and tell Codex not to report staged media as repository sources. Explicitly exclude `.codex/ask-images` from `_existing_sources`. Convert image download, signature, and actual-size failures into the existing user-visible `Repository question failed` path, record only safe aggregate image counts in audit metadata, and rely on the existing `finally` block’s worktree cleanup plus recursive ask-job removal so successful, failed, cancelled, and partially downloaded jobs delete all image bytes.
   - Likely files: `src/telegram_project_manager/bots/ask_manager/prompts.py`, `src/telegram_project_manager/bots/ask_manager/service.py`, `src/telegram_project_manager/bots/ask_manager/images.py`
5. **Cover multimodal behavior** — Expand ask-manager fakes so the Telegram bot can return controlled image bytes or download errors. Test command forwarding and pre-queue validation; successful single-image and album staging; preserved attachment order and MIME-derived extensions; `image_paths` passed to Codex; empty `image_paths` for text-only asks; acknowledgment image counts; rejection of unsupported MIME types, image count, declared limits, actual total limits, invalid signatures, and Telegram download failures; exclusion of staged images from rendered sources; and cleanup after both success and failure. Add a Telegram parsing regression using an `/ask` photo/album caption, while retaining the existing Codex adapter test that converts paths to `LocalImageInput`.
   - Likely files: `tests/test_ask_manager.py`, `tests/test_telegram_bot.py`, `tests/test_code_manager.py`
6. **Document compatibility and rollout** — Update the command reference and repository-question documentation to show `/ask <question>` with optional photo, image document, or album attachments and list the supported formats and limits. Note that the feature uses the existing Codex plan model and Telegram polling attachment parser, requires no new settings or schema migration, and rolls out with existing deployments after application restart. Provider-side image rejection should surface through the normal failed-question reply without affecting subsequent jobs.
   - Likely files: `README.md`

## Risks

- Telegram attachment metadata can omit or understate file sizes; download-time byte limits and signature checks must remain authoritative rather than trusting `IncomingAttachment.file_size`.
- Staged images live inside the temporary checkout so the Codex read-only sandbox can access them; prompts and source filtering must prevent `.codex/ask-images` from being misrepresented as repository evidence.
- Animated or otherwise provider-incompatible GIF files may pass local validation but still be rejected by the configured Codex provider; this must produce a sanitized failed-question response and complete cleanup.
- Albums can approach the 20 MB aggregate limit and occupy an ask concurrency slot during downloads; validation, bounded `download_file` reads, and the existing semaphore limit resource usage.
- A process crash can leave an ask-job directory on disk, matching the existing detached-worktree crash behavior; normal success, failure, cancellation, and shutdown paths must still remove staged media deterministically.
