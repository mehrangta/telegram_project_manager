# Codex plan for mehrangta/telegram_project_manager#14

Job: `c-7f124b6f` · Revision: 2

Implement `/issues` as a durable, topic-aware live view with the authorized 1.A behavior: every successful invocation sends a new issue-list message, immediately makes that message the sole auto-refresh target for its chat or forum topic, and leaves earlier issue-list messages visible but permanently static. One background service refreshes current targets every 60 seconds, grouping targets by repository so GitHub is queried once per repository per cycle. Each issue keeps its linked number and title and gains a visible native copy control for a repository-qualified `/code owner/repository#<issue-number>` command, with `/code #<issue-number>` used only if the qualified command exceeds Telegram's copy-text limit. Existing repository validation, topic isolation, 20-issue cap, most-recently-updated ordering, HTML escaping, and disabled link previews remain unchanged.

## Risks

- Polling every 60 seconds creates sustained GitHub CLI and API traffic. Grouping all current targets by repository limits traffic to one query per distinct repository per cycle, but deployments tracking many repositories may still require a longer interval in a future configurable setting.
- A new `/issues` publication can race with an in-progress refresh. Per-scope locks, a current-row re-read, and conditional database updates are required together; omitting any of them could allow an older message to keep updating or let an old cycle overwrite the new target's hash.
- Telegram may reject edits because a message was deleted, permissions changed, or the message is inaccessible. Permanent and transient errors are not reliably represented by a dedicated exception type, so classification should be centralized and covered with representative Bot API error strings; unknown errors should default to transient retention rather than destructive tracker removal.
- Sending a message and persisting its ID cannot be one transaction. If persistence fails after the send, best-effort deletion may also fail and leave a static orphan message; the previous tracked message must remain authoritative so the system never runs two live targets for one scope.
- The persisted target can refer to a repository that is no longer the scope's active repository. Repository-qualified copied commands prevent targeting the wrong repository, and the message heading identifies its repository; the target remains live until superseded unless the repository is removed from the allowlist.
- Twenty issue rows create twenty native copy buttons and visible commands. This is within the existing issue cap, but rendering tests must enforce Telegram's 4096-character message limit and 256-character copy-text limit, including unusually long allowed repository names and escaped titles.
- Recovery makes subscriptions durable across deployments. A faulty scheduler could repeatedly call external services after every restart, so startup must create only one task, unchanged hashes must suppress Telegram edits, disallowed repositories must be pruned, and shutdown must reliably cancel the loop.
