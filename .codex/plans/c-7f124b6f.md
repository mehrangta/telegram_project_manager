# Codex plan for mehrangta/telegram_project_manager#14

Job: `c-7f124b6f` · Revision: 1

Implement `/issues` as a durable, topic-aware live view. Under the recommended behavior, each successful invocation sends a fresh issue-list message, marks it as the only auto-refresh target for that chat or topic, and supersedes the previous target without deleting the older message. A background service refreshes the latest target every 60 seconds, batching GitHub reads by repository. Each listed issue retains its linked number and title and adds a visible, natively copyable `/code #<issue-number>` command. Existing repository validation, 20-issue limit, updated-descending ordering, HTML escaping, and disabled link previews remain compatible.

## Steps

Give it explicit `Database`, `TelegramBotApi`, and `GhIssueReader` dependencies plus an injectable refresh interval for deterministic tests. Keep repository selection and permission checks in `CommitManager`; the service accepts an already validated chat ID, topic ID, and repository. Render both populated and empty successful results as `OutgoingMessage` values so an empty repository remains live and can display newly opened issues later.

## Risks

- A 60-second poll per repository adds sustained GitHub API and `gh` process load. Grouping subscriptions by repository limits this to one list request per distinct active repository per cycle, but installations with many repositories may still need a longer cadence later.
- Telegram may reject edits when a message was deleted, permissions changed, markup became invalid, or the message is otherwise inaccessible. Permanent message errors must clear only the matching tracker, while transient failures must not silently disable refresh.
- A refresh cycle can overlap a new `/issues` invocation. Without message-ID compare-and-swap checks, the old cycle could edit or clear the newly selected message; conditional database operations and a final tracker re-read are required.
- Twenty titles plus visible `/code` commands and HTML entities approach Telegram's 4096-character limit. Existing title normalization and the 20-item cap should keep displayed text within bounds, but rendering tests should assert the final message remains valid and use the existing truncation policy if necessary.
- Persisted trackers survive deployment, so an implementation bug could repeatedly call GitHub or Telegram after restart. Recovery must validate repository allowlisting, use one scheduler task, and support clean cancellation.
- Native `copy_text` buttons depend on the Telegram Bot API behavior already used elsewhere in this repository. The textual `<code>/code #ID</code>` fallback keeps the command manually copyable if a client does not expose the native control.
