# Codex plan for mehrangta/telegram_project_manager#8

Job: `c-811b8967` · Revision: 2

Add an admin-only `/queue` command that shows all Codex work for the current Telegram chat or exact forum topic, combining `/code` implementation phases with `/ask` repository questions. The view will separate work that is currently running from work waiting for an execution slot, identify every visible item, avoid misleading global-capacity totals, and leave existing `/code`, `/code status`, and `/ask` behavior unchanged.

## Risks

- Code jobs are persisted while `/ask` questions are intentionally in-memory; after a restart, interrupted asks cannot appear in `/queue` because the existing service does not resume them.
- Current chat/topic isolation can produce an empty or short queue view while hidden work in other scopes consumes the services' global concurrency or outstanding limits; the output must not imply global availability.
- Persisted code statuses identify queue membership but do not provide a contractual asyncio semaphore position. Entries should be ordered deterministically without displaying numbered execution positions or estimated start times.
- Ask metadata must be registered, transitioned, and removed at precise lifecycle boundaries. Removing it only in the task callback would incorrectly show failure reporting and workspace cleanup as active Codex execution.
- Status collections used by transitions, admission counting, and snapshots can drift as new phases are added; centralizing and directly testing them is required for long-term accuracy.
- Question text is user-controlled and potentially long or HTML-like; it must be normalized, truncated, and passed through the existing safe response formatter before Telegram delivery.
